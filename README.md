# 法說會逐字稿語調分析系統 (Tone Analysis System)

這是一個專為分析台灣上市公司法人說明會（Conference Call）逐字稿所設計的語調分析系統。本系統結合了本地字典規則（Dictionary-based）與 Google Gemini 大語言模型（AI-based）的優勢，能自動識別發言者角色、定位簡報與問答階段、精煉不確定性語意，並計算出精準的語調指標（Tone Scores）。

本專案提供兩種使用介面：
1. **📊 互動式網頁儀表板 (`index-v2.html`)**：具備高視覺化圖表、拖放式佇列分析、與實時進度條。
2. **⚡ 高效批次命令行工具 (`batch_process.js`)**：專為處理數千份歷史逐字稿而設計，具備自動重試與指數退避機制，極速狂飆。

---

## 🌟 主要功能與特點

* **發言者角色自動分類 (Speaker Role Classification)**: 自動區分「管理層 (Management)」與「其他/主持人 (Others)」，避免非管理層的客套詞彙干擾語調指標。
* **簡報與問答分段定位 (Presentation & Q&A Segmentation)**: 自動辨識簡報階段（Phase P）與開放提問階段（Phase Q），分別計算語調，提供多維度的研究數據。
* **不確定性語意細分精煉 (Uncertainty Refinement)**: 本地字典標記為不確定（UNC）的句子，透過 AI 結合前後文語意進一步細分為偏向正向（POS）或偏向負向（NEG），提高分析靈敏度。
* **批次打包與節流優化 (Batch API Optimization)**: 將大量句子打包成批次（Batch Chunks）發送至 Gemini API，並具備 429 頻率限制自動退避與重試機制，徹底壓低 API 呼叫次數與成本。
* **前端 DOM 虛擬化列表 (Virtual List Rendering)**: 網頁端在載入數千筆資料時能保持 60 FPS 流暢度，絕不凍結網頁。

---

## 🔬 分析流程完整說明 (Full Analysis Pipeline)

本節詳細記錄整套系統從「拿到一份逐字稿文字檔」到「算出 Tone_P / Tone_Q / Tone_T 三個分數」之間，每一個處理步驟的實際邏輯。網頁版 (`index.html` / `index-v2.html`) 與命令行版 (`batch_process.js`) 的核心演算法完全相同，唯一差異在於 AI 呼叫的分批策略與進度顯示方式。

### 總覽：六個處理階段 (Pass)

整批檔案是「先全部做完純字典分析，再統一打包送 AI」，不是每份檔案各自即時呼叫 AI，理由詳見下方「AI 批次呼叫架構」一節。

| Pass | 內容 | 是否呼叫 AI |
|---|---|---|
| 1 | 讀檔、切句、發言人角色判斷（字典規則） | 否 |
| 2 | 批次角色分類 AI（只處理字典判不出來的句子） | 是（可選） |
| 3 | 簡報／問答分段判斷（字典規則，需要已解完的角色） | 否 |
| 4 | 批次問答起點 AI（只處理關鍵字判斷失敗的檔案） | 是（可選） |
| 5 | 計算統計數據、收集不確定（UNC）句子 | 否 |
| 6 | 批次 UNC 方向精煉 AI（整批只呼叫幾次） | 是（可選） |

若未輸入 API Key，Pass 2、4、6 全部跳過，純粹用字典規則產生結果（`AI` 欄位顯示「字典」）。

### 1. 文字前處理與切句 (Text Preprocessing & Sentence Splitting)

**Step 1.1 — 找出逐字稿正文起點**
先用換行 `\n` 把整份文字拆成行，逐行檢查是否符合：
```js
const TRANSCRIPT_START_PAT = /^#{1,3}\s*(transcript|逐字稿)/i;
```
第一個符合的行號記為 `startLine`，正文從**這一行的下一行**開始算（該標題行本身被丟棄）。如果整份文字都沒有這種標題（`startLine` 恆為 0），則從頭開始，不做任何裁切。

> 注意：這裡刻意**只認「transcript／逐字稿」開頭的標題**，不把 `## Q&A` 這種標題也當作正文起點——否則若逐字稿只有 `## Q&A` 標題、沒有 `## Transcript`，會把整段簡報內容誤判為「開場白」而被切掉。

**Step 1.2 — 去除雜訊字串**
在切句之前，先用正則清掉以下三種常見雜訊（整份文字，含裁切後的正文）：
```js
text.replace(/\[\d+:\d+:\d+\]|Page \d+ of \d+|\[\d+ 分鐘\]/gi, '')
```
分別對應：時間戳記（如 `[00:12:34]`）、PDF 分頁殘留（如 `Page 3 of 12`）、以及分鐘標記（如 `[5 分鐘]`）。

**Step 1.3 — 依標點與換行切句**
```js
text.split(/[。！？\n\r]/)
```
用「句號、驚嘆號、問號、換行」四種字元切分，每個片段就是一句。切完之後：
```js
.map(s => s.trim())
.filter(s => s.length > 3 && !/^\d+$/.test(s))
```
去除頭尾空白後，**丟棄長度 ≤3 個字元的片段**（避免標點殘留、單字雜訊被當成一句），也**丟棄純數字的片段**（例如頁碼殘留「12」）。

若切完一句都沒有（整份文字空白或全被濾掉），該檔案直接視為合法的「全零結果」（Tone 全部為 0，不算錯誤），不進入後續任何處理。

**Step 1.4 — 標記 Q&A 起點行**
每一句再檢查是否符合手動或自動的問答起點標記格式：
```js
const QA_START_PAT = /^#{1,3}\s*(q&a|問答|問答環節|提問|analyst\s*q)|^#QA_START/i;
```
符合的句子記為 `isMarker[i] = true`。這種「標記行」本身不是逐字稿內容（它是標題或使用者手動插入的 `#QA_START`），**不會被算進最終的分析結果**，只用來觸發後面的分段判斷（見第 3 節），最後在 Pass 3 結束後的過濾步驟中會被整個丟棄。

使用者可以在逐字稿任何一行手動插入 `#QA_START`，這一行會被立即視為問答段開始（不受第 3 節的最短句數限制）。

### 2. 發言人角色判斷 (Speaker Role Classification)

每一句都要判斷是「管理層 (role=1)」還是「非管理層 (role=0，如主持人/分析師/投資人)」，只有 `role===1` 的句子才會被算進最終的情緒統計——這是為了避免主持人的客套話、分析師的提問內容污染語調分數。

判斷依**優先順序**執行，一旦某一步驟判斷出結果就停止，不會再往下走：

**Step 2.1 — 解析 `**姓名 - 公司, 職稱**` 格式標籤（最可靠）**
```js
const labelPat = /\*\*[^*]+-[^*]+,\s*([^*]+)\*\*/;
```
抓出標籤中的「職稱」部分，比對下方兩份關鍵字清單。

**Step 2.2 — 解析單獨的 `**職稱**` 粗體標籤**
```js
const boldPat = /\*\*([^*]+)\*\*/;
```
同樣比對職稱清單。

以上兩步都是拿抓出來的「職稱字串」比對：
```js
// 管理層職稱關鍵字
const MGMT_TITLES = /總經理|執行長|CEO|CFO|COO|CTO|副總|董事長|董事|首席|主管|協理|經理|財務長|發言人|副發言人|管理層|會計主管|業務|資深/;
// 非管理層關鍵字（用於已抽出的職稱字串比對，可以用較寬鬆的清單）
const NON_MGMT_TITLES = /Investor\s*Relation|IR|主持人|司儀|媒體|分析師|投資人|記者|聽眾|研究員|主持|座談|法說|證券|投顧/;
```

**Step 2.3 — 句子本身含明確職稱（沒有粗體標籤時的次要判斷）**
依序檢查三種規則：
```js
// (a) 句首「稱謂+職稱+早安/午安/你好」→ 判非管理層
//     （這是別人在跟該職稱的人打招呼，例如分析師提問前的開場白「王經理，早安」）
const GREETING_TO_TITLE_PAT = /^.{0,6}(總經理|執行長|CEO|CFO|COO|CTO|副總|董事長|經理|協理|財務長|發言人)[，,、]\s*(早安|午安|晚安|您好|你好|大家好)/;

// (b) 句子含管理層職稱關鍵字（整句比對，不限職稱標籤）→ 判管理層
if (MGMT_TITLES.test(sent)) return 1;

// (c) 句首有明確的發言人標籤前綴（如「主持人：」）→ 判非管理層
const NON_MGMT_LABEL_PREFIX = /^(主持人|司儀|分析師|投資人|法人|媒體|記者|Investor\s*Relation|IR)[:：]/i;
```
> 這裡刻意**只用句首標籤前綴**（而非整句關鍵字）判斷非管理層，因為「法說」「媒體」「投資人」「證券」這類詞管理層自己講話時也常提到，若整句比對會誤判，還會透過下一步的上下文延續機制擴散到後面整份逐字稿。

**Step 2.4 — 上下文延續（沿用前一句角色）**
若以上都判斷不出來，且前一句角色已知，直接沿用前一句的角色。

**Step 2.5 — 真的無法判斷 → 留給 AI 或後續 fallback**
回傳 `null`，交給批次 AI 角色分類（若有 API Key）或最終的 fallback 規則。

**AI 輔助角色分類（有 API Key 才會執行）**
把所有檔案中角色為 `null` 的句子彙整成一份全域清單，連同它前後 2 句「已知是管理層」的句子當作上下文（不含前後所有句子，只挑管理層的句子當參考，控制 prompt 長度），一次性打包送給 Gemini 判斷。回傳格式為 `{"F<檔案序號>S<句子序號>": 0或1}` 的 JSON 物件。

**AI 判斷不到或失敗時的 fallback（`resolveRemainingRoles`）**
逐句掃過去，用「沿用前一個已知角色」補上剩下的 `null`；若前 20 句以內沿用到的是「非管理層」，強制改判為管理層（開場白通常是管理層先發言，避免整份被上下文延續機制誤判成非管理層開頭）。

> 註：曾經嘗試過「連續超過 5 句非管理層就強制翻成管理層」的規則，但實測真實語料發現這會把問答段裡分析師的長篇提問整段誤判成管理層發言、嚴重污染 Tone_Q，因此**沒有採用**這條規則。

**極端情況救援（`finalizeMetrics` 內）**
如果一份檔案跑完以上所有步驟後，**完全沒有任何一句被判為管理層**（且句數 > 5），視為角色判斷整個失敗，啟動救援機制：先把所有簡報段（Phase P）的句子都改判為管理層；如果這樣做完仍然一句管理層都沒有，改成「除了明顯是問答句型（提問關鍵字、`投資先進/法人/分析師/主持人/Q:` 這類標籤）以外，全部視為管理層」。

### 3. 簡報／問答分段判斷 (Presentation vs. Q&A Phase Segmentation)

每一句最終要標記為 Phase P（簡報段，`phase=0`）或 Phase Q（問答段，`phase=1`），且**一旦進入問答段就不會再切回簡報段**（單向轉換）。

**判斷順序（`analyzePhases`）**：

1. **手動標記行**（第 1.4 節的 `isMarker`）：只要遇到，立即切換為問答段，不受任何最短句數限制。
2. **自動關鍵字判斷**：只在句子索引 `idx >= QA_MIN_SENT`（目前設為 20）之後才開始檢查，避免逐字稿一開頭就被誤判進入問答段。符合以下任一條件即觸發：
   ```js
   const QA_START_PAT   = /^#{1,3}\s*(q&a|問答|問答環節|提問|analyst\s*q)|^#QA_START/i;
   const QA_TEXT_PAT     = /接下來是\s*(Q&A|問答|意見交換|提問時間)|進入\s*(Q&A|問答環節|提問環節|意見交換環節)|^開放提問|Q&A\s*(時間|環節|開始)|question[\s-]*(and|&)[\s-]*answer\s*session|open (the floor|the call) for questions|we('|’)?ll now (take|open for) questions|let'?s (start|begin) the Q&A|operator instructions|now open for questions|this concludes our (key messages|prepared remarks|presentation)/i;
   const QA_LABEL_PAT     = /^(投資先進|法人|分析師|主持人|Q)[:：]/;
   // 或者「Q&A 概念詞」與「轉場/開始詞」同句共現：
   const QA_CONCEPT_PAT    = /Q\s*&?\s*A|問答|提問|意見交(換|流)|問與答/i;
   const QA_TRANSITION_PAT = /接下來|現在|開始|進行|進入|即將|最後|來到|下一(個|個階段)|下個|準備|歡迎|開放|階段/;
   ```
3. **預告句過濾**：如果句子含「延後詞」（之前、準備、稍後、等一下、待會、結束後、報告後、簡報後、結束之後、分為、分成），代表這通常只是「預告等一下會有 Q&A」的簡報段句子，**不算真正的轉場**，除非句子裡**同時**出現「立即開始詞」（現在/正式/馬上/立刻/即刻、或「接下來／接著／接續」緊接著「是/進入/進行/開始」再緊接 Q&A 概念詞、或「一併/逐一/先回答」、或「針對…提問/問題…回答」），才視為真正的轉場而不套用過濾。
   > 這是為了處理「我們現在開始進行 Q&A 時間，我們會將**之前**蒐集到的提問先進行回答」這種句子——它同時有延後詞（之前）又是真轉場，如果只看延後詞會被誤判成預告句而漏掉真正的問答起點。
4. **輔助訊號**：非管理層的提問句（含 `請問/想請教/請教一下/請教/想問/可否請教/想了解一下/請問一下` 任一詞）之後，緊接著管理層回答，這種「非管理層問→管理層答」的組合**連續出現 2 次**，也視為已進入問答段。

**關鍵字判斷完全失敗時的 AI 輔助（有 API Key 才會執行）**：如果整份逐字稿用以上規則都判斷不出任何問答起點（且句數 > 20），才把該檔案送去給批次 AI 找轉場句，避免每份檔案都呼叫 AI。AI 回傳格式為 `{"<檔案序號>": "F<檔案序號>S<句子序號>"}`，找不到轉折點的檔案不會出現在回傳結果裡（這是合法結果，不算 AI 失敗）。

### 4. 字典情緒分類 (Dictionary-based Sentiment Classification)

每一句用四份字典分別做**字串包含比對**（不是完整詞語比對，也沒有中文斷詞）：

| 類別 | 字數 | 意義 |
|---|---|---|
| POS（正向） | 206 | 正面營運/財務用語 |
| NEG（負向） | 370 | 負面營運/財務用語 |
| LIT（法律/爭議） | 301 | 法律訴訟、監管、爭議相關用語 |
| UNC（不確定） | 138 | 語氣不確定、有待觀察的用語 |

```js
function matches(sent, dict) {
  return [...new Set(dict.filter(w => w && sent.includes(w)))];
}
```
每份字典都算出「命中詞去重後的清單」，該類別的命中「次數」就是清單長度（同一個詞在同一句裡出現多次只算一次）。

**分類規則（`localCat`）**：先檢查是否命中中性排除清單（見第 5 節），若是直接歸類 `NEUTRAL`；否則取四個類別中**命中詞數量最多**的類別。**如果有並列最高分（例如 POS 和 NEG 命中數一樣多），該句會同時歸入這兩個類別**，在統計時對兩邊的分母都各計一次（刻意設計，非 bug）。若四個類別都是 0 命中，歸類為 `NONE`（不計入任何統計，但仍計入「總句數」）。

> **字典內容的已知限制**：LIT 字典原本是依照財經情緒字典（Loughran-McDonald 式的 Litigious 類別）翻譯建置，這類字典在正式書面文件（如英文 10-K 年報）裡運作良好，但直接套用到口語逐字稿時，部分被翻譯進來的常見連接詞（例如「對」「第」「主要」「應」等）在日常口語中出現頻率極高，會造成大量與法律爭議無關的誤判。經過使用真實語料驗證（2429 份逐字稿抽樣），移除 33 個此類泛用詞後，LIT 誤判句數減少 92.1%，全語料平均 Tone_T 從 -0.1127 修正為 +0.2917。
>
> NEG 字典裡的「問題」也有類似但更細緻的問題：這個字本身有歧義，可以指「疑問/提問」（中性，法說會問答段最常見的用法）也可以指「產品/技術上的問題」（負向），字典比對分不出來，一律算負向。實測 500 份真實語料抽樣：「問題」是所有負向命中裡出現次數最多的詞（2373 次），其中 1967 句（占全部負向判定的 17.1%）完全靠「問題」單獨判定，抽樣檢視這些句子大多是「回答問題」「開放提問」這類問答格式用語，不是真的在講負面業務問題。跟「對／第／主要」不同的是，「問題」**確實有合理的負向用法**（例如「資安問題」「品質問題」），不是完全零訊號的詞，所以沒有直接刪除，而是**改成只在特定複合詞組出現時才算負向**：把單獨的「問題」從 NEG 字典移除，改成加入 `資安問題、品質問題、技術問題、財務問題、安全問題、法律問題、環保問題、庫存問題、匯率問題、資金問題、稅務問題、供應問題、良率問題、效率問題、製程問題、法規問題、發生問題、出現問題、遇到問題、造成問題` 這 20 個具體詞組。
>
> 過程中也發現兩個候選詞組因為容易誤判而被排除：**「有問題」**實測有 42.9% 的命中其實是「沒有問題」「沒什麼問題」這種否定句型（意思是「沒問題」，反而不是負向），字串比對抓不到否定詞，加了反而更容易誤判；**「出問題」**會被「提出問題」（中性的「提問」）這種常見詞組字串包含，實測誤判率也很高，因此都沒有採用。
>
> 拿全部 2429 份真實語料重新驗證：NEG 誤判句數減少 19.2%（53,252→43,035 句），全語料平均 Tone_T 從 +0.2917 修正為 +0.3281，2429 份裡有 1954 份分數往上調、只有 17 份小幅往下調（往下調的都是因為移除「問題」後跟其他類別打平、被同時歸類到兩個類別的正常併列效果，不是誤判）。
>
> 若後續要調整字典內容，建議先用類似方法（統計某個字「單獨造成分類」的次數、抽樣檢查句子、必要時檢查否定詞與常見詞組包含關係）驗證影響範圍，再決定是否移除或改成複合詞組。

### 5. 中性排除清單 (Neutral Exclusion List)

在字典分類**之前**，每句話會先檢查是否符合 103 條排除規則（`DEFAULT_EXCLUSIONS`），符合的話直接歸類 `NEUTRAL`、不列入正負向統計（但仍計入「排除句數」）。規則分七大類：

| 分類 | 條數 | 範例 |
|---|---|---|
| `greeting`（問候語） | 34 | 「大家好」「謝謝各位的聆聽」「各位投資先進」 |
| `serial`（序號條列） | 11 | 「第一點」「（一）」`^\d+[.、。]` 開頭的數字序號 |
| `time`（時間日期） | 7 | 「今天」「本季」「Q1」～「Q4」 |
| `presentation`（簡報導言） | 17 | 「法說會」「聊天室」「舉手功能」「邀請…說明」 |
| `closing`（結束語） | 6 | 「到此結束」「法說會…結束」「告一段落」 |
| `qa_host`（Q&A 主持詞） | 18 | 「QA 時間」「是否還有其他提問」「第 X 個問題」 |
| `transition`（轉場銜接） | 10 | 「接下來報告」「藉此機會」「時間戳記」`^\(\d{2}:\d{2}:\d{2}\)` |

每條規則有四種比對類型：`exact`（完全相等）、`prefix`（開頭符合）、`contains`（包含）、`regex`（正規表示式）。使用者可以在網頁版介面自行新增/刪除排除規則（`+ 新增排除詞`／每條規則的 ✕ 按鈕），或按「重置」恢復預設清單。

### 5.5 為什麼分數普遍偏正向，以及對應的修正

實測拿真實的 2429 份語料跑純字典分析（不含 AI）發現：**97.2% 的檔案 Tone_T 是正向、只有 1.6% 負向**，平均 Tone_T = 0.328。這裡面有一部分是合理的——法說會逐字稿是管理層主導、經過設計的溝通場合，習慣用「成長、提升、擴大」包裝，即使講風險也常是正面框架，全語料 POS:NEG 命中比例 3.5:1 這個量級跟財經文本分析文獻對法說會逐字稿的觀察大致吻合。但實測也挖出兩個會系統性把分數推向正面、確實需要修正的問題：

1. **完全沒有否定詞處理**：字典比對原本是單純子字串比對（`sent.includes(w)`），看不懂「沒有」「未」「不」。實測 POS 命中裡有 2.00% 緊接在否定詞後面（例如「並未達到高點」被算成正向），NEG 只有 1.13% 被否定——兩邊都有、但 POS 誤判比例明顯更高。修正：新增 `NEGATION_MARKERS`，字典詞緊鄰在否定詞後面時，直接不算進該類別（視為中性）。**沒有**嘗試把否定詞判斷「反轉」到相反類別，因為沒有通用規則能保證翻轉方向一定正確（例如「沒有風險」偏正向、「不會成長」偏負向，方向不一致），保守只做排除。
   > **已知例外**：「不動產」（房地產類資產）恰好是「不」+ LIT 字典詞「動產」（可移動財產）黏成的固定複合詞，兩者是對比詞、不是「動產被否定」，實測全語料有 703 句因此被誤排除。已加入 `NEGATION_FALSE_POSITIVES` 例外清單排除這個誤判；之後如果實測發現同類狀況（否定詞+字典詞剛好黏成另一個不相關複合詞），可以繼續往這個清單加。
2. **一句話同時有褒有貶時，詞數少的一方訊號被整句丟棄**：`localCat()` 原本是「哪個類別在這句命中詞數最多，整句就算那一類」。實測全語料有 8.10%（3.4 萬句）的句子同時命中 POS 與 NEG 詞（「雖然...但是...」這種句型），其中詞數多的一方整句通吃、另一方被完全丟棄——而 POS 通吃掉 NEG 訊號的次數是反過來（NEG 通吃 POS）的 2.7 倍，這是另一個把分數推向正面的機制。修正：只要同一句 POS、NEG 都有命中（不管誰的詞數比較多），兩邊都保留、讓 `calcTone` 的分子自然相抵，不再是詞數多的一方贏家全拿。

**修正後的實測影響**（全語料 2429 份，兩項修正一起套用）：平均 Tone_T 從 0.328 降到 0.285（下降約 13%），86.6%（2102/2429）檔案的分數下降、僅 6.5% 上升，15 份從「正向」被修正為「非正向」、12 份反向修正。97.2%→97.1% 的整體正向檔案佔比幾乎沒變（這個二元門檻本來就對「普遍性下修」不敏感，中位數/平均數的下降才是真正反映修正幅度的指標），代表法說會逐字稿本身偏正向的性質依然存在——這部分是預期中的產業/文本類型效應，不是 bug；但兩項修正確實移除了一部分不該存在的系統性正面偏誤。網頁版與 `batch_process.js` 都採用同一套邏輯。

> **後續追加的字典內容清理**：上面兩項是判斷邏輯的修正，這裡另外處理字典內容本身的問題，實測抽樣後確認：
> * **POS 字典的「好」是雜訊主要來源**：全語料有 23691 句含「好」，但抽樣後發現大多數是「好，謝謝」「好的，我們...」這類口語發語詞／應答語助詞，或「大家好」這種問候語，真正表達「好=正面評價」的比例很低。移除這個裸字，改用精確度高很多的複合詞「很好」（1151 句）、「非常好」（715 句）、「良好」（1949 句）取代——這幾個複合詞在抽樣裡幾乎都是真正的正面評價語句。原本已存在的「好轉」「更好」不受影響。
> * **POS 字典的「需」同樣是雜訊**：全語料有 28687 句含「需」，抽樣後發現絕大多數是「需要」「需求」「所需」這類中性的描述性用詞（描述有什麼需求，不代表對公司是好是壞），直接移除、不做替換。
> * **「穩定」同時出現在 POS 與 NEG 兩份字典裡**：字面上「穩定」是偏正面／中性的詞（穩定成長、穩健的投資策略等），出現在 NEG 字典裡是誤收錄；而且在加了否定詞偵測之後，「不穩定」（真正的負面訊號）反而會因為「穩定」被否定排除掉、完全抓不到負面訊號。修正：把「穩定」從 NEG 字典移除（只留在 POS），改成專門收錄複合詞「不穩定」到 NEG 字典（全語料 234 句），精確捕捉「不穩定」的負面含義，不再依賴否定詞排除機制硬猜。
>
> **這次字典清理的實測影響**（在前面兩項邏輯修正之後再疊加測量）：正向檔案佔比從 97.1% 降到 **94.5%**，平均 Tone_T 從 0.285 降到 **0.258**；76%（1854/2429）檔案分數下降，只有 19% 上升；66 份從「正向」被修正為「非正向」，僅 4 份反向修正——比前兩項邏輯修正的效果更明顯、方向也更一致。**三項修正（否定詞處理＋混合句訊號保留＋字典內容清理）疊加起來，整體平均 Tone_T 從最初的 0.328 降到 0.258（下降約 21%）**，法說會逐字稿偏正向的產業效應依然存在（94.5% 仍是正向），但系統性的偏誤已經大幅收斂。網頁版與 `batch_process.js` 都採用同一份字典。

### 6. 不確定句（UNC）方向精煉 (Uncertainty Refinement)

被歸類為 UNC 的句子，除了計入「不確定句數」，還會再判斷語意方向偏正向還是偏負向：

**字典規則（預設，一律先算一次）**：
```js
function uncRefine(sent) {
  if (/(成長|擴張|增長|提升|擴大|發展|上升)/.test(sent)) return 1; // 偏正向
  if (/(風險|壓力|困難|不確定|危險|挑戰|下滑)/.test(sent)) return 2; // 偏負向
  return 0; // 依然中性
}
```

**AI 精煉（有 API Key 才會執行）**：把所有檔案的 UNC 句子彙整成一份全域清單，一次打包送給 Gemini，請它根據語意（不只看關鍵字，例如「負債比下降」要判斷為正向，即使沒出現「成長」二字）重新判斷方向，回傳 `{"F<檔案序號>D<明細序號>": 0/1/2}`。AI 判斷成功的句子會覆蓋字典規則的結果並標記 `aiRefined: true`；AI 沒回應到或整批失敗的句子，維持字典規則算出的方向。

> **UNC 精煉的「整輪重試」機制**：UNC 句子的方向判斷原則上都應該真的經過 AI 分析，不能因為暫時性的額度／頻率限制就輕易放棄回字典。除了每個 chunk 內部自己的重試（見上方「AI 批次呼叫架構」），UNC 精煉還多一層機制：一輪全部 chunk 跑完後，把還沒拿到結果的句子收集起來，等 15 秒（讓 RPM/TPM 這種滾動視窗有機會恢復），再重新打包送一次，最多再重試 3 輪（`UNC_SWEEP_MAX`），才真的放棄、fallback 回字典。中途任何一輪的失敗都不會提早標記成 `aiError`，只有用盡所有重試輪次、句子依然拿不到結果，才會誠實回報失敗。網頁版與命令行版（`batch_process.js`）都採用同一套機制。

> **每日額度 (RPD) 用完時的快速失敗機制**：上面「整輪重試」機制是為了因應 RPM/TPM 這種**幾秒到一分鐘內就會恢復**的滾動視窗限制而設計的——但如果 429 錯誤其實是**當天的請求數 (RPD) 已經整個用完**，那不管重試幾次、每輪等 15 秒、總共等 3 輪，都不可能恢復，只會等到太平洋時間午夜額度重置才行。實測真實批次時發現：一旦當天 RPD 已經用盡，2429 份檔案的批次會在**每一個** chunk 上都重複「4 次重試＋3 輪整批重試」的完整流程，每次都必然失敗，等於是把數十分鐘甚至數小時的時間浪費在注定失敗的等待上，結果畫面上還是幾乎全部顯示「字典」——這正是使用者反覆回報「怎麼還是顯示字典」「不確定句應該都要套 AI」的真正成因之一。
>
> Google 的 429 錯誤內容（`error.details[].violations[]`）會標明是哪一種額度被打到：RPD 用完的 `quotaId` 固定包含 `"PerDay"`（例如 `"GenerateRequestsPerDayPerProjectPerModel-FreeTier"`），跟 RPM/TPM 的 `quotaId`（含 `"PerMinute"`）明確不同。新增的 `isDailyQuotaError()` 會解析這個欄位，一旦偵測到是 RPD 用完：
> * 立刻設定全域旗標 `dailyQuotaExhausted = true`，**不等待、不重試**，直接讓這個 chunk 判定失敗。
> * 之後（同一次執行內）所有 AI 呼叫的重試迴圈、UNC 的整輪重試迴圈、chunk 間的固定間隔，全部提早跳過，改成立刻 fallback 回字典——不再對外發送任何請求，讓整批次能快速跑完，而不是卡在注定失敗的等待上。
> * 畫面上會出現一則持續顯示的警告橫幅（「今日 Gemini API 每日請求額度 (RPD) 已用完...」），跟原本會被後續階段狀態覆蓋掉的 `ai-status` 分開顯示，讓使用者能明確知道「這不是程式壞掉，是額度真的用完了，要等重置」，而不是誤以為又是某種隱藏的 bug。
> * 每次按下「開始分析」都會重新把旗標歸零，不會影響下一次全新的執行。

> **修正：`onChunkDone` 誤把「chunk 有句子失敗」當成「chunk 整批失敗」**：UNC 精煉每個 chunk 通常橫跨多份檔案、多個句子，`chunkFailed`（見上方「整輪重試」）是整個 chunk 層級的旗標——只要 chunk 裡有任何一句在最後一輪仍拿不到結果就是 `true`。舊版程式碼一看到 `chunkFailed` 就直接把這個 chunk 涵蓋的**所有**句子都標記失敗、丟棄結果，即使 `chunkResultMap` 裡其實已經包含了同一個 chunk 內其他句子的正確 AI 判斷值——等於把明明成功拿到的 AI 結果也一起當失敗處理掉。現在改成逐句判斷：每一句先看自己的 label 有沒有出現在 `chunkResultMap` 裡，有就套用、標記 `aiRefined`；沒有的話才依 `chunkFailed` 決定要不要算這一句真正失敗。

> **新增 `aiAnySuccess`，取代範圍過窄的 `details.some(d=>d.aiRefined)`**：三態徽章原本用「明細裡有沒有任何一句 `aiRefined:true`」來判斷「部分 AI」，但這個欄位**只有 UNC 精煉階段會設**——如果一份檔案是角色分類或問答起點判斷這兩個階段部分成功、部分失敗（跟 UNC 精煉完全無關，甚至這份檔案可能根本沒有 UNC 句子），畫面上仍然會顯示「字典」，即使這份檔案其實真的吃到了一部分 AI 判斷。改成三個 AI 階段（角色分類／問答起點／UNC 精煉）只要對某份檔案有任一次真正成功，就設定該檔案的 `aiAnySuccess = true`，`aiBadge()` 改讀這個涵蓋全部階段的旗標，「部分 AI」判斷不再只看得到 UNC 這一個階段。

> **`clearQueue()`／`clearAll()` 加上 `processing` 守衛**：這兩個函式原本沒有檢查是否正在批次執行中，如果使用者在 `runBatch()` 跑到一半時按下「清除佇列」或「全部清除」，`queue`／`results` 陣列會被立刻換成新的空陣列，但 `runBatch()` 內部迴圈仍持著跑批次開始時就算好的索引（`pending[k]`、`queue[f.qIdx]` 等）繼續存取，下一次存取就會拿到 `undefined` 而整個批次連帶炸掉、卡死。現在這兩個函式在 `processing===true` 時直接不做任何事。

> **`runBatch()` 加上 `try/finally`**：六個 Pass 中如果有任何一個拋出沒被內部 try/catch 接住的例外（原本各 Pass 各自有自己的 try/catch，但仍可能有意料之外的漏網例外），函式會直接中斷在半途，`processing` 永遠卡在 `true`、「開始分析」按鈕永遠停用、進度徽章永遠顯示執行中，使用者只能重新整理整個頁面才能再跑一次。現在把收尾（重設 `processing`、重畫畫面、恢復按鈕）移進 `finally` 區塊，不管中途是否拋出例外都保證會執行。

### 7. Tone 分數計算公式

```js
function calcTone(s) {
  const p = s.pos + s.pos_u;              // 正向 = 字典正向 + UNC偏正向
  const n = s.neg + s.neg_u;              // 負向 = 字典負向 + UNC偏負向
  const l = s.lit;                        // 法律/爭議
  const d = s.pos + s.neg + s.lit + s.unc; // 分母 = 正向+負向+法律+不確定（原始命中數，不含 pos_u/neg_u 重複計）
  return d === 0 ? 0 : (p - n - l) / d;
}
```
* **分子**：（正向 + UNC偏正向）－（負向 + UNC偏負向）－ 法律/爭議。法律/爭議類別視同負向訊號，直接從分子扣除。
* **分母**：正向 + 負向 + 法律/爭議 + 不確定的**原始命中句數**（不論最後被精煉成正向還是負向，都只算一次）。`NEUTRAL`（中性排除）與 `NONE`（無命中）不計入分母。
* 分別對 Phase P（簡報段）與 Phase Q（問答段）各自的統計數字代入，得到 `Tone_P` 與 `Tone_Q`；再用 P、Q 兩段合併後的統計數字代入，得到整體的 `Tone_T`。

### 8. AI 批次呼叫架構 (Batched AI Call Architecture)

**設計動機**：早期版本是「每份檔案各自即時呼叫 AI」，處理數千份檔案時，光是角色判斷、問答起點、UNC 精煉三種需求疊加起來，可能就是幾千次 API 呼叫，免費額度（例如某些模型每日僅 20 次）幾乎瞬間用完，而且序列等待每次網路回應的時間會直接疊加成整批次的總耗時。

**改進做法**：全部檔案先做完 Pass 1（字典分析），把「需要 AI 幫忙」的項目（不確定角色的句子、找不到 Q&A 起點的檔案、UNC 句子）彙整成一份**全域清單**，用 `F<檔案序號>S<句子序號>`（角色/問答起點）或 `F<檔案序號>D<明細序號>`（UNC）這種全域唯一編號當 key，再依文字量切成**盡量少的幾個大請求**一次送出（`BATCH_CHUNK_CHARS` 控制每個請求的輸入字元數上限），讓一次 API 回應就能同時解掉多個檔案的問題。

**失敗容錯設計**：
* 每個 chunk 的失敗只影響**這個 chunk 涵蓋的檔案**，不會波及其他 chunk（已成功的結果照樣套用）。
* AI 判斷不到、或整批失敗的項目，一律 fallback 回字典規則的結果，並誠實標記 `aiError`。
* Q&A 起點找不到轉折點屬於**合法結果**（並非每份逐字稿都一定有問答段），不算 AI 失敗。

**AI 徽章的三種狀態**：因為同一份檔案的 UNC 句子可能被切進不同 chunk，可能出現「這份檔案有些句子成功套用 AI、有些因為別的 chunk 失敗而 fallback 回字典」的混合情況。畫面上的 AI 欄位（結果表格與明細視窗標題旁）因此有三種狀態，而不是只有「有/沒有」二選一：
* **✓ AI**（綠色）：`aiUsed && !aiError`，這份檔案的 AI 判斷全部成功，沒有任何失敗。
* **部分 AI**（紅色）：`aiUsed && aiError`，但明細裡至少有一句真的被 AI 精煉過（`details.some(d=>d.aiRefined)`）——代表這份檔案**有一部分**吃到 AI，另一部分因為某個 chunk 失敗而 fallback 回字典，兩者混在同一份結果裡。
* **字典**（橘色）：其餘情況，包含完全沒開 AI、或這份檔案涉及的所有 AI 呼叫都失敗。

> **明細視窗裡「▲ 修正正向」「▼ 修正負向」標記的常見誤解**：這個標記在 `refine>0`（不確定句有判斷出方向）時就會顯示，**不代表一定是 AI 判斷的**——UNC 句子在 Pass 5（純字典階段）就已經用 `uncRefine()` 關鍵字規則（看到「成長/提升」猜正向、「風險/下滑」猜負向）猜過一次方向，這個字典猜測結果本來就會顯示標記；只有真正被 AI 覆蓋過的句子才會額外標示「・AI 已確認」，字典猜測的標成「・字典猜測（未套用 AI）」，兩種樣式（顏色、框線）也刻意做出區別。實測發現使用者容易誤以為「看到修正標記＝這句被 AI 處理過」，進而懷疑「檔案裡明明有修正標記、上面卻顯示字典徽章」是 bug——但其實只是把字典猜測跟 AI 確認搞混，徽章本身的判斷是正確的。

**網頁版的提早揭露機制**：Pass 5（角色與問答段都確定、統計數據算完）之後，分數就會立刻寫進結果表格顯示出來——這時候的分數已經是完整、有效的結果（字典規則＋角色/問答 AI，只差 UNC 精煉還沒套用），不用等到 chunk 數量最多、AI 額度用完時重試最久的 Pass 6（UNC 精煉）整個跑完才看得到任何東西。

Pass 6 本身也不是「全部 chunk 跑完才一次套用」：`batchRefine` 每個 chunk 一結束（不管成功或失敗）就會透過回呼通知，只更新**這個 chunk 實際涵蓋到的檔案**、重新計算這些檔案的 Tone 分數、立刻重畫結果表格，而不是等所有 chunk 都跑完才統一寫入一次。因為 chunk 是依「總字元數」跨檔案連續切的（不是每個檔案各自一個 chunk），同一份檔案的 UNC 句子有可能被切進不同 chunk，所以會看到同一筆結果的精煉進度隨著 chunk 陸續完成而分階段增加，不是一次跳到最終值。就算某個 chunk 或整個 Pass 6 失敗，已經顯示、已經更新過的分數也不會被清空或蓋掉。

**重試會讀取 Google 建議的等待秒數**：三個 AI 函式（`batchClassifyRoles`／`batchFindQAStart`／`batchRefine`，網頁版與 CLI 皆同）遇到 429／5xx 時，都會優先解析 Google 回應中的建議重試秒數（`RetryInfo.retryDelay` 或訊息裡的「retry in Xs」）來等待，而非用固定的指數退避時間硬猜。

> **已修正的問題**：網頁版原本三個 AI 函式各自有獨立的重試邏輯，只有 `batchRefine` 曾經加過解析建議秒數，`batchClassifyRoles` 跟 `batchFindQAStart` 一直都是用固定退避（4 次重試總共只等約 12 秒）。這會造成一個容易被誤判成「額度用完」的現象：**同一批次裡，有的 chunk 成功、有的又失敗，散布沒有規律**，看起來不像額度用盡那種「用完後全部失敗」的乾淨分界。真正原因是 TPM／RPM 這類「每分鐘」的滾動視窗限制——如果 Google 建議等 30-40 秒，但程式只硬等 12 秒就放棄重試，剛好卡在視窗還沒恢復的 chunk 就會永久失敗，晚幾秒送出、視窗剛好過了的 chunk 則會成功，因而呈現散布、忽有忽無的樣子。三個函式現在統一改成解析並等待建議秒數（上限 65 秒），大幅降低這種因為重試等太短而錯失視窗恢復時機的情況。

**沒有任何「一次失敗就永久關閉整個批次 AI」的全域開關**——每個 chunk 獨立重試、獨立 fallback，一份檔案或一個 chunk 的失敗不會影響後面數千份檔案的 AI 判斷機會。CLI 版每個 chunk 之間固定間隔 1 秒、網頁版固定間隔 2 秒，降低一開始就撞到頻率限制的機率。

> **已修正的分組問題**：早期版本用「固定檔案數」切 chunk（角色分類固定每 40 份檔案一組、Q&A 起點判斷固定每 10 份檔案一組），跟這些檔案實際有多少不確定內容完全脫鉤。實測發現：單一檔案測試永遠沒事，但真實跑數千份檔案的大批次會頻繁失敗——因為剛好有幾組的 40 份檔案裡不確定角色的句子特別多，組出來的 JSON 回應長度超過 `maxOutputTokens` 上限被硬生生截斷，導致 `JSON.parse` 失敗，看起來就像「AI 判讀失敗」。現已改成跟網頁版一致的**依總字元數動態切 chunk**（見上方 `packIntoChunks`／`BATCH_CHUNK_CHARS`），內容量大的自動分成更多、更小的 chunk；Q&A 起點判斷也改成只送 `QA_MIN_SENT`（第 20 句）之後的內容（更早的句子本來就不可能是轉場點），並將角色分類的 `maxOutputTokens` 由 8192 提高到 32768（此請求的輸出量會隨 chunk 內不確定句數量增加）。

**目前使用的模型**：固定寫死 `gemini-flash-lite-latest`（Google 官方 `-latest` 別名，不指定固定版本號、不開放使用者選擇），避免像先前使用的 `gemini-1.5-flash` 那樣，模型被 Google 除役後整個工具的 AI 功能無預警失效。

> **每日額度用量評估**：這個模型的免費層額度不是只有「每日請求數（RPD）」一種限制，Google 用量儀表板（https://ai.dev/rate-limit）同時追蹤 RPM（每分鐘請求數）、**TPM（每分鐘輸入 token 數）**、RPD 三種限制，超過任何一種都會被拒絕。UNC 精煉原本用「固定 400／100 句一批」（網頁版／CLI 版）切 chunk，跟句子實際長度脫鉤，真實語料一次批次通常有十幾萬句 UNC，這個階段本身就要 300+ 次請求。改成跟其他兩個 AI 階段一致的**依總字元數動態切 chunk**後，一度把 chunk 上限設到 10 萬字元換取最少的請求次數，但實測發現這樣單一 chunk 太大，會讓**短時間內送出的 token 總量**超過 TPM 上限（曾實測到 374.1K，遠超過 250K）——單靠壓低 RPD 不夠。
>
> 最終取得平衡：chunk 上限調整為 **6 萬字元**（`BATCH_CHUNK_CHARS`），並在三個 AI 階段之間都加上**固定 2 秒的 chunk 間隔**（原本網頁版的角色分類、Q&A 起點判斷這兩個階段完全沒有間隔，是 TPM 超標的主因之一）。拿全部 2429 份真實語料重新實測：角色分類 50 個 chunk、Q&A 起點判斷 86 個 chunk、UNC 精煉 109 個 chunk，**三項合計約 245 次請求**，比原本的 145 次多，但仍在 500 次/日的額度內有充裕餘裕，同時最大單一 chunk 字元數降低約 4 成，大幅降低短時間內衝高 TPM 的風險。

### 9. 資安、可中止性與崩潰復原

**XSS 防護**：檔名、逐字稿句子、手動貼上的排除清單標籤全部是使用者可控內容，畫面上多處用 `innerHTML` 直接拼接顯示。同一頁面又明文放著 Gemini API Key（輸入欄位＋`localStorage`），代表一次 XSS 就足以連 Key 一起被偷走。新增 `escapeHtml()`，凡是把使用者可控字串塞進 `innerHTML` 的地方（結果表格的公司名稱、明細視窗標題與逐句內容、排除清單標籤、失敗訊息）一律先跳脫。

**API Key 不再放在 URL query string**：原本 `?key=${apiKey}` 會被瀏覽器擴充功能的請求記錄器、企業 proxy 日誌、devtools 的 HAR 匯出檔等非預期管道記錄下來。三個 AI 函式都改成用 Google 官方支援的 `x-goog-api-key` request header 傳遞 Key。

**「中止本次分析」**：批次規模一大（2000+ 份檔案）又疊加 chunk 內重試與 UNC 整輪重試，最壞情況可能卡上數十分鐘，使用者原本除了整頁重新整理（會遺失所有尚未寫出的進度）之外沒有其他脫身手段。新增 `userAborted` 旗標，用法跟 `dailyQuotaExhausted` 快速失敗機制完全一致：所有重試/等待迴圈都會檢查它並提早跳出。按下「中止」只是設旗標，不用 `AbortController` 硬砍正在等待中的 fetch——已經送出、正在等回應的那個請求還是會正常拿到結果並套用，只是不會再送出新的請求。

**Google 建議的重試秒數不再被武斷上限蓋過**：`computeWaitMs()` 只在「沒有 `RetryInfo`、只能用固定指數退避硬猜」時才套用 65 秒上限；如果 Google 明確建議等 90 秒，就照實等 90 秒，不會為了套上限而提早重試、賭一次幾乎必敗的請求白白燒掉重試次數。三個 AI 函式重複的「429 且判定為當日額度用完」邏輯也抽成共用的 `checkQuotaExhausted()`。

**崩潰保護（分析結果備份/還原）**：佇列裡的檔案（`File` 物件、尚未上傳的內容）無法序列化進 `localStorage`，沒辦法整批復原；但已經算出來的 `results`（字典＋AI 分析分數）是純資料，備份成本低、也是使用者最不想遺失的部分。`backupResults()` 只在 Pass 5 剛寫入初步結果、以及整批次結束時這兩個時機點寫入 `localStorage`（不在 Pass 6 每個 chunk 都寫，避免大批次下頻繁 `JSON.stringify` 造成效能負擔或撞到儲存配額——`persistSet` 內部已有 try/catch，配額爆掉時安靜地不寫入，不會讓分析中斷）。頁面重新載入時如果偵測到殘留的備份，會顯示一個橫幅讓使用者選擇「恢復結果」或「忽略」。

**大批次下 UNC 精煉的重畫節流**：Pass 6 原本每個 chunk 完成就整表重畫一次，chunk 數上百上千時會造成卡頓。改成套用跟 Pass 1 一致的 `RENDER_MIN_INTERVAL` 節流間隔，分數本身仍然每個 chunk 都即時重算，只有「整表重繪」這個較貴的 DOM 操作被節流。

**CSV 摘要報表新增 AI 可信度欄位**：摘要報表最常被往外分享（給管理層、IR），使用者如果不點開逐句明細，看不出這份報表的 Tone 分數有多少比例只是字典猜測。新增 `AI_Status`（AI完成／部分AI／字典）與 `UNC_AI_Confirmed_Pct`（不確定句裡有多少比例真的拿到 AI 判斷）兩欄，把資料可信度直接寫進報表本身。

### 10. 補套用 AI（只重新處理還沒真正拿到 AI 判斷的不確定句）

**設計動機**：當天 RPD 額度用完、使用者中途按了中止、或單純某個 chunk 剛好失敗，都會讓一部分不確定句停在「字典猜測」狀態。過去唯一的解法是把整批（可能 2000+ 份檔案）從頭重新上傳、重新解析一次——不但浪費時間，也會把已經成功拿到 AI 判斷的句子重新送一次，白白消耗當天好不容易恢復的額度。

**做法**：`reapplyAI()` 直接掃描目前的 `results[]`，透過每筆結果自帶的 `uncIndices`／`details[].aiRefined` 找出所有「這句是不確定句、但還沒真正被 AI 覆蓋過」的句子，只把**這些句子**重新打包送給 `batchRefine()`——不重新上傳檔案、不重新跑 Pass 1-5、已經成功的句子完全不會被重新請求或覆蓋。套用的更新邏輯（逐 chunk 即時更新畫面、逐句判斷成功/失敗、`aiUsed`/`aiError`/`aiAnySuccess` 旗標）跟 `runBatch()` Pass 6 完全一致，只是直接操作 `results[i]` 本身，不透過 `runBatch()` 內部那個跑完就消失的 `files` 陣列。

**使用時機**：額度在太平洋時間午夜重置後、或懷疑有殘留的字典 fallback 想再試一次 AI 時，直接按結果表格上方的「補套用 AI（僅未確認句）」，不用重新選取任何檔案。旁邊會即時顯示「還有 N 句不確定句未套用 AI」，數字歸零時代表全部不確定句都已經拿到真正的 AI 判斷。如果目前沒有任何未確認的句子，按下去只會跳出提示，不會發送任何請求。

> **修正：一旦有過失敗就永遠卡在「部分 AI」，補完也回不去「AI 完成」**：角色分類／問答起點這兩個階段用 `roleAiError`／`qaAiError` 這兩個 sticky 旗標記錄失敗（沒有對應的補救機制，一旦失敗就是失敗）。但 UNC 精煉不一樣——它有「補套用 AI」可以事後修好，如果也用一個 sticky 的 `uncAiError` 旗標，補完之後這個旗標永遠不會被清掉，會讓明明已經 100% 拿到 AI 判斷的檔案卡在「部分 AI」，永遠顯示不出「✓ AI」。修正後 `hasAiError()` 不再依賴一個要手動清除的旗標，而是每次都直接檢查 `uncIndices` 裡還有沒有任何一句 `details[].aiRefined` 是 `false`——「補套用 AI」補完最後一句的當下，這裡自然就會算出「沒有 gap 了」，不需要額外寫程式碼去重設任何東西。

**中止鍵改成真的立刻生效**：舊版的「中止本次分析」只設一個旗標，讓迴圈在下一次檢查點自然跳出，但**不會**主動打斷正在等待中的那個 fetch——如果剛好卡在等 Google 回應，最壞情況要等到 `GEMINI_TIMEOUT_MS`（90 秒）逾時才會真的停下來，使用者等不到那麼久就會覺得「這個按鈕根本沒用」。新增 `activeAbortControllers` 集合追蹤所有還在飛行中的請求，`abortBatch()` 現在會直接呼叫每一個的 `.abort()`，讓等待中的請求立刻用 `AbortError` 失敗、被上層重試迴圈接住並因為 `userAborted` 已經是 `true` 而不再重試（重試前的等待也一併跳過，不會白等）。按鈕本身也會立刻變成「中止中...」並停用，讓使用者確定自己的點擊真的有生效，不用乾等著猜。

> **修正：「部分 AI」點了「補套用 AI」卻說沒事做，看起來自相矛盾**：「部分 AI」其實有兩種完全不同的成因——(1) 還有不確定句沒套用 AI（`uncGap`，「補套用 AI」修得好），(2) 角色分類或問答起點判斷失敗（`roleAiError`／`qaAiError`，「補套用 AI」修不了，因為那兩個階段需要的原始逐字稿內容——`allSents`／`isMarker`／`roles`——只存在於 `runBatch()` 執行期間的暫時 `files` 陣列，批次跑完就不會留在 `results[]` 裡，沒有東西可以重送）。畫面上這兩種成因共用同一個「部分 AI」徽章、沒有區分，使用者看到後直覺點「補套用 AI」，卻被告知「沒有需要補套用的句子」，以為是新 bug。修正：
> * `aiBadge()` 的 tooltip 依實際成因分開講清楚——是不確定句沒套用（可以補）、還是角色/問答判斷失敗（不能補，需重新上傳該檔案完整重跑）、還是兩者都有。
> * `reapplyAI()` 在「沒有不確定句需要補套用」時，如果還有檔案卡在角色/問答判斷失敗，會額外講清楚有幾份檔案、為什麼「補套用 AI」幫不上忙，而不是只回一句聽起來像「一切正常」的訊息。

> **修正：明細視窗打開後不會自動重畫，導致畫面卡在舊快照、跟實際資料矛盾**：明細視窗（`renderModal()`）原本只有在 `openModal()` 打開的那一刻畫一次，之後如果背景的 AI 更新了這份檔案的資料（`runBatch()` Pass 6 某個 chunk 完成，或按了「補套用 AI」），視窗畫面完全不會跟著更新。實測發現：使用者開著某份檔案的明細，看到某句顯示「字典猜測（未套用 AI）」；接著按「補套用 AI」，這句其實已經被成功修好了，但視窗畫面停留在打開當下的舊快照，還是顯示「字典猜測」——而這時再按一次「補套用 AI」，因為那句真的已經修好了，會正確回報「沒有需要補套用的句子」。兩個畫面同時看起來就像互相矛盾、像是 bug，但其實只是明細視窗沒有跟著背景更新重畫。
>
> 修正：新增 `refreshModalIfOpen(ri)`，在 `runBatch()` Pass 6 與 `reapplyAI()` 每次更新完受影響的檔案後呼叫。判斷「是不是目前開著的這份檔案」刻意用**索引**比對，不是物件參照比對——因為 `runBatch()` Pass 6 的 `recalcAndUpdate` 是用 `results[i] = {...}` 整個換掉新物件（不是原地修改），如果用物件參照比較，這種情況會抓不到是同一份檔案，而 `reapplyAI()` 的 `recalcAndUpdate` 又是原地修改同一個物件——兩種寫法都要正確處理，索引比對兩種情況都適用；重畫前也會重新從 `results[]` 抓一次 `detailResult`，才能拿到換過的新物件。

## 🛠️ 使用說明

### 1. 視覺化網頁儀表板 (`index-v2.html`)

* **開啟方式**：直接雙擊 `index-v2.html` 用 Chrome 或 Safari 等瀏覽器開啟；或使用本機網頁伺服器開啟：

## 📊 數據輸出格式

批次分析完成後，系統會自動在指定位置生成兩份 UTF-8 (含 BOM) 格式的報表：

### 1. 摘要總報表 (`Tone_Summary.csv`)
每家公司/每次法說會輸出為單一行，包含：
* `TONE_P` / `TONE_Q` / `TONE_T`：簡報段、問答段、以及整體的語調分數。
* `P_正向`、`P_負向`、`Q_正向`、`Q_負向` 等各類情緒有效句數量。
* `AI`：標記該檔案是否有成功調用 AI 進行深度輔助分析。

### 2. 逐句分析明細表 (`Tone_Detail.csv`)
包含所有檔案中被拆分出來的每一句話的詳細分類：
* 該句子的原始文字、句子所屬角色（管理層/其他）、句子所屬階段（簡報/問答）。
* 本地字典匹配到的類別（POS/NEG/LIT/UNC/NEUTRAL/NONE）及對應的情緒詞。
* AI 針對不確定（UNC）句子的精煉結果（POS-Refined / NEG-Refined / 依然中性）。
