# 法說會逐字稿語調分析系統 (Tone Analysis System)

這是一個專為分析台灣上市公司法人說明會（Conference Call）逐字稿所設計的語調分析系統。本系統結合了本地字典規則（Dictionary-based）與 Google Gemini 大語言模型（AI-based）的優勢，能自動識別發言者角色、定位簡報與問答階段、精煉不確定性語意，並計算出精準的語調指標（Tone Scores）。

本專案提供兩種使用介面：
1. **📊 互動式網頁儀表板 (`index-v2.html`)**：具備高視覺化圖表、拖放式佇列分析、與實時進度條。
2. **⚡ 高效批次命令行工具 (`batch_process.js`)**：專為處理數千份歷史逐字稿而設計，具備自動重試與指數退避機制，極速狂飆。

---

## 🌟 主要功能與特點

* **發言者角色自動分類 (Speaker Role Classification)**: 自動區分「管理層 (Management)」與「其他/主持人 (Others)」，避免非管理層的客套詞彙干擾語調指標。
* **簡報與問答分段定位 (Presentation & Q&A Segmentation)**: 自動辨識簡報階段（Phase P）與開放提問階段（Phase Q），分別計算語調，提供多維度的研究數據。
* **不確定性語意細分精煉 (Uncertainty Refinement)**: 本地字典標記為不確定（UNC）的句子，透過 AI 結合前後文語意進一步細分為偏向正向（POS）或偏向負向（NEG），提高分析靈敏度。
* **批次打包與節流優化 (Batch API Optimization)**: 將大量句子打包成批次（Batch Chunks）發送至 Gemini API，並具備 429 頻率限制自動退避與重試機制，徹底壓低 API 呼叫次數與成本。
* **前端 DOM 虛擬化列表 (Virtual List Rendering)**: 網頁端在載入數千筆資料時能保持 60 FPS 流暢度，絕不凍結網頁。

---

## 🔬 分析流程完整說明 (Full Analysis Pipeline)

本節詳細記錄整套系統從「拿到一份逐字稿文字檔」到「算出 Tone_P / Tone_Q / Tone_T 三個分數」之間，每一個處理步驟的實際邏輯。網頁版 (`index.html` / `index-v2.html`) 與命令行版 (`batch_process.js`) 的核心演算法完全相同，唯一差異在於 AI 呼叫的分批策略與進度顯示方式。

### 總覽：六個處理階段 (Pass)

整批檔案是「先全部做完純字典分析，再統一打包送 AI」，不是每份檔案各自即時呼叫 AI，理由詳見下方「AI 批次呼叫架構」一節。

| Pass | 內容 | 是否呼叫 AI |
|---|---|---|
| 1 | 讀檔、切句、發言人角色判斷（字典規則） | 否 |
| 2 | 批次角色分類 AI（只處理字典判不出來的句子） | 是（可選） |
| 3 | 簡報／問答分段判斷（字典規則，需要已解完的角色） | 否 |
| 4 | 批次問答起點 AI（只處理關鍵字判斷失敗的檔案） | 是（可選） |
| 5 | 計算統計數據、收集不確定（UNC）句子 | 否 |
| 6 | 批次 UNC 方向精煉 AI（整批只呼叫幾次） | 是（可選） |

若未輸入 API Key，Pass 2、4、6 全部跳過，純粹用字典規則產生結果（`AI` 欄位顯示「字典」）。

### 1. 文字前處理與切句 (Text Preprocessing & Sentence Splitting)

**Step 1.1 — 找出逐字稿正文起點**
先用換行 `\n` 把整份文字拆成行，逐行檢查是否符合：
```js
const TRANSCRIPT_START_PAT = /^#{1,3}\s*(transcript|逐字稿)/i;
```
第一個符合的行號記為 `startLine`，正文從**這一行的下一行**開始算（該標題行本身被丟棄）。如果整份文字都沒有這種標題（`startLine` 恆為 0），則從頭開始，不做任何裁切。

> 注意：這裡刻意**只認「transcript／逐字稿」開頭的標題**，不把 `## Q&A` 這種標題也當作正文起點——否則若逐字稿只有 `## Q&A` 標題、沒有 `## Transcript`，會把整段簡報內容誤判為「開場白」而被切掉。

**Step 1.2 — 去除雜訊字串**
在切句之前，先用正則清掉以下三種常見雜訊（整份文字，含裁切後的正文）：
```js
text.replace(/\[\d+:\d+:\d+\]|Page \d+ of \d+|\[\d+ 分鐘\]/gi, '')
```
分別對應：時間戳記（如 `[00:12:34]`）、PDF 分頁殘留（如 `Page 3 of 12`）、以及分鐘標記（如 `[5 分鐘]`）。

**Step 1.3 — 依標點與換行切句**
```js
text.split(/[。！？\n\r]/)
```
用「句號、驚嘆號、問號、換行」四種字元切分，每個片段就是一句。切完之後：
```js
.map(s => s.trim())
.filter(s => s.length > 3 && !/^\d+$/.test(s))
```
去除頭尾空白後，**丟棄長度 ≤3 個字元的片段**（避免標點殘留、單字雜訊被當成一句），也**丟棄純數字的片段**（例如頁碼殘留「12」）。

若切完一句都沒有（整份文字空白或全被濾掉），該檔案直接視為合法的「全零結果」（Tone 全部為 0，不算錯誤），不進入後續任何處理。

**Step 1.4 — 標記 Q&A 起點行**
每一句再檢查是否符合手動或自動的問答起點標記格式：
```js
const QA_START_PAT = /^#{1,3}\s*(q&a|問答|問答環節|提問|analyst\s*q)|^#QA_START/i;
```
符合的句子記為 `isMarker[i] = true`。這種「標記行」本身不是逐字稿內容（它是標題或使用者手動插入的 `#QA_START`），**不會被算進最終的分析結果**，只用來觸發後面的分段判斷（見第 3 節），最後在 Pass 3 結束後的過濾步驟中會被整個丟棄。

使用者可以在逐字稿任何一行手動插入 `#QA_START`，這一行會被立即視為問答段開始（不受第 3 節的最短句數限制）。

### 2. 發言人角色判斷 (Speaker Role Classification)

每一句都要判斷是「管理層 (role=1)」還是「非管理層 (role=0，如主持人/分析師/投資人)」，只有 `role===1` 的句子才會被算進最終的情緒統計——這是為了避免主持人的客套話、分析師的提問內容污染語調分數。

判斷依**優先順序**執行，一旦某一步驟判斷出結果就停止，不會再往下走：

**Step 2.1 — 解析 `**姓名 - 公司, 職稱**` 格式標籤（最可靠）**
```js
const labelPat = /\*\*[^*]+-[^*]+,\s*([^*]+)\*\*/;
```
抓出標籤中的「職稱」部分，比對下方兩份關鍵字清單。

**Step 2.2 — 解析單獨的 `**職稱**` 粗體標籤**
```js
const boldPat = /\*\*([^*]+)\*\*/;
```
同樣比對職稱清單。

以上兩步都是拿抓出來的「職稱字串」比對：
```js
// 管理層職稱關鍵字
const MGMT_TITLES = /總經理|執行長|CEO|CFO|COO|CTO|副總|董事長|董事|首席|主管|協理|經理|財務長|發言人|副發言人|管理層|會計主管|業務|資深/;
// 非管理層關鍵字（用於已抽出的職稱字串比對，可以用較寬鬆的清單）
const NON_MGMT_TITLES = /Investor\s*Relation|IR|主持人|司儀|媒體|分析師|投資人|記者|聽眾|研究員|主持|座談|法說|證券|投顧/;
```

**Step 2.3 — 句子本身含明確職稱（沒有粗體標籤時的次要判斷）**
依序檢查三種規則：
```js
// (a) 句首「稱謂+職稱+早安/午安/你好」→ 判非管理層
//     （這是別人在跟該職稱的人打招呼，例如分析師提問前的開場白「王經理，早安」）
const GREETING_TO_TITLE_PAT = /^.{0,6}(總經理|執行長|CEO|CFO|COO|CTO|副總|董事長|經理|協理|財務長|發言人)[，,、]\s*(早安|午安|晚安|您好|你好|大家好)/;

// (b) 句子含管理層職稱關鍵字（整句比對，不限職稱標籤）→ 判管理層
if (MGMT_TITLES.test(sent)) return 1;

// (c) 句首有明確的發言人標籤前綴（如「主持人：」）→ 判非管理層
const NON_MGMT_LABEL_PREFIX = /^(主持人|司儀|分析師|投資人|法人|媒體|記者|Investor\s*Relation|IR)[:：]/i;
```
> 這裡刻意**只用句首標籤前綴**（而非整句關鍵字）判斷非管理層，因為「法說」「媒體」「投資人」「證券」這類詞管理層自己講話時也常提到，若整句比對會誤判，還會透過下一步的上下文延續機制擴散到後面整份逐字稿。

**Step 2.4 — 上下文延續（沿用前一句角色）**
若以上都判斷不出來，且前一句角色已知，直接沿用前一句的角色。

**Step 2.5 — 真的無法判斷 → 留給 AI 或後續 fallback**
回傳 `null`，交給批次 AI 角色分類（若有 API Key）或最終的 fallback 規則。

**AI 輔助角色分類（有 API Key 才會執行）**
把所有檔案中角色為 `null` 的句子彙整成一份全域清單，連同它前後 2 句「已知是管理層」的句子當作上下文（不含前後所有句子，只挑管理層的句子當參考，控制 prompt 長度），一次性打包送給 Gemini 判斷。回傳格式為 `{"F<檔案序號>S<句子序號>": 0或1}` 的 JSON 物件。

**AI 判斷不到或失敗時的 fallback（`resolveRemainingRoles`）**
逐句掃過去，用「沿用前一個已知角色」補上剩下的 `null`；若前 20 句以內沿用到的是「非管理層」，強制改判為管理層（開場白通常是管理層先發言，避免整份被上下文延續機制誤判成非管理層開頭）。

> 註：曾經嘗試過「連續超過 5 句非管理層就強制翻成管理層」的規則，但實測真實語料發現這會把問答段裡分析師的長篇提問整段誤判成管理層發言、嚴重污染 Tone_Q，因此**沒有採用**這條規則。

**極端情況救援（`finalizeMetrics` 內）**
如果一份檔案跑完以上所有步驟後，**完全沒有任何一句被判為管理層**（且句數 > 5），視為角色判斷整個失敗，啟動救援機制：先把所有簡報段（Phase P）的句子都改判為管理層；如果這樣做完仍然一句管理層都沒有，改成「除了明顯是問答句型（提問關鍵字、`投資先進/法人/分析師/主持人/Q:` 這類標籤）以外，全部視為管理層」。

### 3. 簡報／問答分段判斷 (Presentation vs. Q&A Phase Segmentation)

每一句最終要標記為 Phase P（簡報段，`phase=0`）或 Phase Q（問答段，`phase=1`），且**一旦進入問答段就不會再切回簡報段**（單向轉換）。

**判斷順序（`analyzePhases`）**：

1. **手動標記行**（第 1.4 節的 `isMarker`）：只要遇到，立即切換為問答段，不受任何最短句數限制。
2. **自動關鍵字判斷**：只在句子索引 `idx >= QA_MIN_SENT`（目前設為 20）之後才開始檢查，避免逐字稿一開頭就被誤判進入問答段。符合以下任一條件即觸發：
   ```js
   const QA_START_PAT   = /^#{1,3}\s*(q&a|問答|問答環節|提問|analyst\s*q)|^#QA_START/i;
   const QA_TEXT_PAT     = /接下來是\s*(Q&A|問答|意見交換|提問時間)|進入\s*(Q&A|問答環節|提問環節|意見交換環節)|^開放提問|Q&A\s*(時間|環節|開始)|question[\s-]*(and|&)[\s-]*answer\s*session|open (the floor|the call) for questions|we('|’)?ll now (take|open for) questions|let'?s (start|begin) the Q&A|operator instructions|now open for questions|this concludes our (key messages|prepared remarks|presentation)/i;
   const QA_LABEL_PAT     = /^(投資先進|法人|分析師|主持人|Q)[:：]/;
   // 或者「Q&A 概念詞」與「轉場/開始詞」同句共現：
   const QA_CONCEPT_PAT    = /Q\s*&?\s*A|問答|提問|意見交(換|流)|問與答/i;
   const QA_TRANSITION_PAT = /接下來|現在|開始|進行|進入|即將|最後|來到|下一(個|個階段)|下個|準備|歡迎|開放|階段/;
   ```
3. **預告句過濾**：如果句子含「延後詞」（之前、準備、稍後、等一下、待會、結束後、報告後、簡報後、結束之後、分為、分成），代表這通常只是「預告等一下會有 Q&A」的簡報段句子，**不算真正的轉場**，除非句子裡**同時**出現「立即開始詞」（現在/正式/馬上/立刻/即刻、或「接下來／接著／接續」緊接著「是/進入/進行/開始」再緊接 Q&A 概念詞、或「一併/逐一/先回答」、或「針對…提問/問題…回答」），才視為真正的轉場而不套用過濾。
   > 這是為了處理「我們現在開始進行 Q&A 時間，我們會將**之前**蒐集到的提問先進行回答」這種句子——它同時有延後詞（之前）又是真轉場，如果只看延後詞會被誤判成預告句而漏掉真正的問答起點。
4. **輔助訊號**：非管理層的提問句（含 `請問/想請教/請教一下/請教/想問/可否請教/想了解一下/請問一下` 任一詞）之後，緊接著管理層回答，這種「非管理層問→管理層答」的組合**連續出現 2 次**，也視為已進入問答段。

**關鍵字判斷完全失敗時的 AI 輔助（有 API Key 才會執行）**：如果整份逐字稿用以上規則都判斷不出任何問答起點（且句數 > 20），才把該檔案送去給批次 AI 找轉場句，避免每份檔案都呼叫 AI。AI 回傳格式為 `{"<檔案序號>": "F<檔案序號>S<句子序號>"}`，找不到轉折點的檔案不會出現在回傳結果裡（這是合法結果，不算 AI 失敗）。

### 4. 字典情緒分類 (Dictionary-based Sentiment Classification)

每一句用四份字典分別做**字串包含比對**（不是完整詞語比對，也沒有中文斷詞）：

| 類別 | 字數 | 意義 |
|---|---|---|
| POS（正向） | 206 | 正面營運/財務用語 |
| NEG（負向） | 351 | 負面營運/財務用語 |
| LIT（法律/爭議） | 301 | 法律訴訟、監管、爭議相關用語 |
| UNC（不確定） | 138 | 語氣不確定、有待觀察的用語 |

```js
function matches(sent, dict) {
  return [...new Set(dict.filter(w => w && sent.includes(w)))];
}
```
每份字典都算出「命中詞去重後的清單」，該類別的命中「次數」就是清單長度（同一個詞在同一句裡出現多次只算一次）。

**分類規則（`localCat`）**：先檢查是否命中中性排除清單（見第 5 節），若是直接歸類 `NEUTRAL`；否則取四個類別中**命中詞數量最多**的類別。**如果有並列最高分（例如 POS 和 NEG 命中數一樣多），該句會同時歸入這兩個類別**，在統計時對兩邊的分母都各計一次（刻意設計，非 bug）。若四個類別都是 0 命中，歸類為 `NONE`（不計入任何統計，但仍計入「總句數」）。

> **字典內容的已知限制**：LIT 字典原本是依照財經情緒字典（Loughran-McDonald 式的 Litigious 類別）翻譯建置，這類字典在正式書面文件（如英文 10-K 年報）裡運作良好，但直接套用到口語逐字稿時，部分被翻譯進來的常見連接詞（例如「對」「第」「主要」「應」等）在日常口語中出現頻率極高，會造成大量與法律爭議無關的誤判。經過使用真實語料驗證（2429 份逐字稿抽樣），移除 33 個此類泛用詞後，LIT 誤判句數減少 92.1%，全語料平均 Tone_T 從 -0.1127 修正為 +0.2917。若後續要調整字典內容，建議先用類似方法（統計某個字「單獨造成分類」的次數）驗證影響範圍，再決定是否移除。

### 5. 中性排除清單 (Neutral Exclusion List)

在字典分類**之前**，每句話會先檢查是否符合 103 條排除規則（`DEFAULT_EXCLUSIONS`），符合的話直接歸類 `NEUTRAL`、不列入正負向統計（但仍計入「排除句數」）。規則分七大類：

| 分類 | 條數 | 範例 |
|---|---|---|
| `greeting`（問候語） | 34 | 「大家好」「謝謝各位的聆聽」「各位投資先進」 |
| `serial`（序號條列） | 11 | 「第一點」「（一）」`^\d+[.、。]` 開頭的數字序號 |
| `time`（時間日期） | 7 | 「今天」「本季」「Q1」～「Q4」 |
| `presentation`（簡報導言） | 17 | 「法說會」「聊天室」「舉手功能」「邀請…說明」 |
| `closing`（結束語） | 6 | 「到此結束」「法說會…結束」「告一段落」 |
| `qa_host`（Q&A 主持詞） | 18 | 「QA 時間」「是否還有其他提問」「第 X 個問題」 |
| `transition`（轉場銜接） | 10 | 「接下來報告」「藉此機會」「時間戳記」`^\(\d{2}:\d{2}:\d{2}\)` |

每條規則有四種比對類型：`exact`（完全相等）、`prefix`（開頭符合）、`contains`（包含）、`regex`（正規表示式）。使用者可以在網頁版介面自行新增/刪除排除規則（`+ 新增排除詞`／每條規則的 ✕ 按鈕），或按「重置」恢復預設清單。

---

### 6. 不確定句（UNC）方向精煉 (Uncertainty Refinement)

被歸類為 UNC 的句子，除了計入「不確定句數」，還會再判斷語意方向偏正向還是偏負向：

**字典規則（預設，一律先算一次）**：
```js
function uncRefine(sent) {
  if (/(成長|擴張|增長|提升|擴大|發展|上升)/.test(sent)) return 1; // 偏正向
  if (/(風險|壓力|困難|不確定|危險|挑戰|下滑)/.test(sent)) return 2; // 偏負向
  return 0; // 依然中性
}
```

**AI 精煉（有 API Key 才會執行）**：把所有檔案的 UNC 句子彙整成一份全域清單，一次打包送給 Gemini，請它根據語意（不只看關鍵字，例如「負債比下降」要判斷為正向，即使沒出現「成長」二字）重新判斷方向，回傳 `{"F<檔案序號>D<明細序號>": 0/1/2}`。AI 判斷成功的句子會覆蓋字典規則的結果並標記 `aiRefined: true`；AI 沒回應到或整批失敗的句子，維持字典規則算出的方向。

### 7. Tone 分數計算公式

```js
function calcTone(s) {
  const p = s.pos + s.pos_u;              // 正向 = 字典正向 + UNC偏正向
  const n = s.neg + s.neg_u;              // 負向 = 字典負向 + UNC偏負向
  const l = s.lit;                        // 法律/爭議
  const d = s.pos + s.neg + s.lit + s.unc; // 分母 = 正向+負向+法律+不確定（原始命中數，不含 pos_u/neg_u 重複計）
  return d === 0 ? 0 : (p - n - l) / d;
}
```
* **分子**：（正向 + UNC偏正向）－（負向 + UNC偏負向）－ 法律/爭議。法律/爭議類別視同負向訊號，直接從分子扣除。
* **分母**：正向 + 負向 + 法律/爭議 + 不確定的**原始命中句數**（不論最後被精煉成正向還是負向，都只算一次）。`NEUTRAL`（中性排除）與 `NONE`（無命中）不計入分母。
* 分別對 Phase P（簡報段）與 Phase Q（問答段）各自的統計數字代入，得到 `Tone_P` 與 `Tone_Q`；再用 P、Q 兩段合併後的統計數字代入，得到整體的 `Tone_T`。

### 8. AI 批次呼叫架構 (Batched AI Call Architecture)

**設計動機**：早期版本是「每份檔案各自即時呼叫 AI」，處理數千份檔案時，光是角色判斷、問答起點、UNC 精煉三種需求疊加起來，可能就是幾千次 API 呼叫，免費額度（例如某些模型每日僅 20 次）幾乎瞬間用完，而且序列等待每次網路回應的時間會直接疊加成整批次的總耗時。

**改進做法**：全部檔案先做完 Pass 1（字典分析），把「需要 AI 幫忙」的項目（不確定角色的句子、找不到 Q&A 起點的檔案、UNC 句子）彙整成一份**全域清單**，用 `F<檔案序號>S<句子序號>`（角色/問答起點）或 `F<檔案序號>D<明細序號>`（UNC）這種全域唯一編號當 key，再依文字量切成**盡量少的幾個大請求**一次送出（`BATCH_CHUNK_CHARS` 控制每個請求的輸入字元數上限），讓一次 API 回應就能同時解掉多個檔案的問題。

**失敗容錯設計**：
* 每個 chunk 的失敗只影響**這個 chunk 涵蓋的檔案**，不會波及其他 chunk（已成功的結果照樣套用）。
* AI 判斷不到、或整批失敗的項目，一律 fallback 回字典規則的結果，並誠實標記 `aiError`，畫面上顯示「字典」而非「✓ AI」，避免使用者誤以為某份檔案有吃到 AI 判斷。
* Q&A 起點找不到轉折點屬於**合法結果**（並非每份逐字稿都一定有問答段），不算 AI 失敗。

**命令行版 (`batch_process.js`) 的額外設計**：因應數千份檔案的大批次會發出遠多於單檔測試的請求量，更容易撞到免費額度的頻率限制（RPM），`callGemini` 會優先解析 Google 回應中的建議重試秒數（`RetryInfo.retryDelay` 或訊息裡的「retry in Xs」）來等待，而非用太短的固定指數退避時間硬闖；每個 chunk 之間固定間隔 1 秒，降低一開始就撞到頻率限制的機率。**沒有任何「一次失敗就永久關閉整個批次 AI」的全域開關**——每個 chunk 獨立重試、獨立 fallback，一份檔案或一個 chunk 的失敗不會影響後面數千份檔案的 AI 判斷機會。

**目前使用的模型**：固定寫死 `gemini-flash-lite-latest`（Google 官方 `-latest` 別名，不指定固定版本號、不開放使用者選擇），避免像先前使用的 `gemini-1.5-flash` 那樣，模型被 Google 除役後整個工具的 AI 功能無預警失效。
